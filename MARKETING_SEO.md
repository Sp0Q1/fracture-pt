### Platform Security Trust Signals

A security company's own platform must be exemplary. These are the security guarantees we market:

- **Tenant Isolation** -- Every query is scoped to the authenticated user's organisation and role. Client A cannot see Client B's data, findings, or engagements. Period. Our architecture is designed to support fully isolated per-client environments (dedicated database and infrastructure) for enterprise customers who require complete separation.
- **Least Privilege by Design** -- Clients see only their own engagements and findings. Pentesters access only their assigned engagements. Administrators manage workflow without accessing other organisations' sensitive data.
- **No IDORs** -- Every resource access is verified against the authenticated user's organisation AND role. We test for this class of vulnerability in our own platform with the same rigour we apply to client engagements.
- **OWASP Top 10 Hardened** -- Our platform is continuously tested against the full OWASP Top 10, including injection, broken access control, cross-tenant data leakage, and privilege escalation.
- **Audit Trail** -- Every action in the platform is logged: scope submissions, offer creation, finding additions, report downloads. Full accountability for clients, pentesters, and administrators.
- **Role-Based Access Control** -- Three distinct roles (Client, Pentester, Administrator) with precisely scoped permissions. No role can exceed its intended access boundaries.

**Marketing angle:** "We sell security -- so our own platform had better be secure. We hold ourselves to the same standard we hold our clients to. Our access controls, tenant isolation, and vulnerability posture are tested continuously with the same methodology we use on your applications."

## 11. Platform Roles & User Journey Messaging

### Role-Based Marketing Messages

The platform serves three distinct user types. Marketing and onboarding copy must speak to each role's needs:

#### Client (Organisation Owner / Member)
**Onboarding message:** "Welcome to GetHacked. Define what needs testing, accept your offer, and track findings in real time -- all from your dashboard."

**Key page elements:**
- Scope wizard with guided forms (targets, IP ranges, domains, testing windows, exclusions, contacts)
- Engagement dashboard showing active and past pentests
- Findings view with severity filtering, category breakdown, and remediation status
- Report download centre (PDF, executive summary, technical details)
- Organisation settings (team members, billing)

**Value message:** "You stay in control. See exactly what is being tested, what has been found, and what you need to fix -- without waiting for a final email."

#### Pentester (Assigned per Engagement)
**Onboarding message:** "You have been assigned to an engagement. Review the scope, add your findings, and deliver results -- all within the platform."

**Key page elements:**
- Engagement view with scope details, testing windows, and client contacts
- Finding submission form: description, technical details, impact, recommendation, severity (Extreme/High/Elevated/Moderate/Low), category
- Engagement status tracking

**Value message (for recruitment/partnerships):** "Work on clearly scoped engagements with structured finding templates. No ambiguity about scope boundaries. Focus on what you do best: finding vulnerabilities."

**Security note:** Pentesters see ONLY the engagements they are assigned to. They cannot browse other clients, other engagements, or platform-wide data.

#### Administrator (Platform Admin)
**Key page elements:**
- Scope review queue (incoming pentest requests)
- Offer builder (pricing, timeline, deliverables)
- Pentester assignment interface
- Platform-wide engagement overview (status, timelines, completion)

**Security note:** Administrators manage workflow but access controls ensure appropriate boundaries between administrative functions and client data.

### User Journey: Client Requesting a Pentest

This journey is the basis for the "How It Works" marketing section and any product demo:

```
1. Client signs up / logs in
2. Client clicks "Request a Pentest"
3. Scope wizard: targets, IP ranges, domains, testing windows, exclusions, contacts
4. Scope is submitted --> appears in admin review queue
5. Admin reviews scope, creates offer (price, timeline, deliverables)
6. Client receives offer in dashboard, can accept or discuss
7. Client accepts --> engagement is created
8. Admin assigns pentester(s) to the engagement
9. Pentesters begin testing within the agreed window
10. Pentesters add findings (description, technical detail, impact, recommendation, severity, category)
11. Client sees findings appear in real time in their dashboard
12. Pentest completes --> findings compile into downloadable report
13. Client downloads professional report (executive summary + technical details)
```

**Marketing takeaway:** Every step happens in-app. No external tools. No email attachments. Full audit trail. This is the core differentiator to emphasise in demos, sales calls, and landing page copy.
